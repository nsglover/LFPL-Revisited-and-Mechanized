\section{Polynomial-Time Completeness}

Now that we have a denotational semantics, we can concretely state what it means for \lfpl to be complete with respect to polynomial-time computation. We focus on polynomial-time computations on the natural number. Since $\tpsem{\tplist{\tpunit}} = \{\ast\}^*$ is the set of finite-length strings from a unary alphabet, it is can be viewed as the set of unary-encoded natural numbers. However, we could just as easily have used the binary representation $\tpsem{\tplist{\tpsum{\tpunit}{\tpunit}}}$ for $\mathbb N$. So, instead of choosing a particular encoding, we can let the ``digits'' of the number be encoded by any type $A$ such that $\dfree{A}$. We require $\dfree{A}$ because plenty of polynomial-time computable functions (even non-size-increasing ones) duplicate the digits of their input, which is impossible in \lfpl if $A$ contains diamonds.

\begin{theorem}[Size-Restricted Polynomial-Time Completeness]
\label{thm:completeness-main}
Let $A$ be a type such that $\dfree{A}$. For every polynomial-time computable function $f : \tpsem{A}^* \to \tpsem{A}^*$ such that $|f(x)| \le |x|$ for all $x \in \tpsem{A}^*$, there exists a closed term $M : \tparr{\tplist{A}}{\tplist{A}}$ such that $\tmsem{M}{\envnil} = f$.
\end{theorem}

This theorem is originally proven in \cite{hofmannStrengthNonsizeIncreasing2002}. The constraint that $f$ does not increase the length of its input is necessary because of the non-size-increasing property; a closed \lfpl function on unit lists has no source of diamonds other than the elements of the input list, so the output list cannot have more diamonds (and therefore more elements) than the input list. As we will see later in \pref{sec:completeness-general-thm}, this constraint is relatively superficial and becomes unnecessary if we don't enforce that the output type of $M$ is an \lfpl list.

Proving \pref{thm:completeness-main} boils down to simulating the polynomial-time Turing machine which computes $f$. This immediately presents two challenges. Say that $P : \mathbb N \to \mathbb N$ is the polynomial function encoding how many steps the Turing machine takes to compute $f$. The first challenge is to simulate the tape of the machine, which for an input list of size $n$ could contain up to $P(n)$ additional values by the time the machine halts. We will only have $n$ diamonds from the input list, yet somehow we have to store up to $P(n)$ additional values; clearly a list will not work for this, since every element of a list must be accompanied by a diamond.

The second challenge is to simulate running the Turing machine for $P(n)$ steps. At first glance, it seems that \lfpl is only really suited for performing $kn$ iterations given $n$ diamonds and some constant $k$. This can be acheived for $k = 1$ by iterating down a list and building it back up during the iteration process, and then it is easy to repeat this for any constant $k$ number of times. This is useful but offers no real insight into how to perform, say, $n^2$ iterations.



\subsection{The Bounded Stack Data Structure}
\label{sec:completeness-bounded-stack-definition}

% TODO: Discuss Hofmann's array data structure briefly at the end of this section? (If not here, then definitely somewhere).
We first address the challenge of simulating the tape of the Turing machine. A common approach to this is to use a data structure consisting of a head element and two stacks, each representing one side of the tape relative to the head element. \comment{Jan}{Do you know a citation for this?} Concretely, in \lfpl, if we use the type $A$ for tape symbols, then the type of our tape data structure would be $\tptens{\tplist{A}}{\tptens{A}{\tplist{A}}}$. The first element represents the section of the tape to the left of the head, the second element represents the symbol at the head of the tape, and the third element represents the right section of the tape. Treating elements of $L(A)$ like stacks, we can shift the head to the left by popping a symbol from the left list and pushing the current head symbol to the right list.

Of course, we cannot actually use the above data structure; given $n$ diamonds, it can only hold $n + 1$ values. Instead, we construct a type $S(A)$ (which may depend on $P$) that supports a stack-like interface, at least as long as it doesn't have to hold more than $P(n)$ elements. Then, we can use something of type $\tptens{S(A)}{\tptens{A}{S(A)}}$ to encode the tape.

We have actually already seen the first key insight into how to construct such a type in \pref{example:susp}. This shows us that we really don't need any diamonds to store data, we just need them to interact with it. Using this idea, we can design our data structure to remain fully suspended by default, requiring zero diamonds to store it. The push and pop functions will have to temporarily borrow some diamonds to interface with the suspended data.

The second key insight is that we do not need to un-suspend the entire data structure to interface with it. This is good, because if the stack held $P(n)$ values, we would still need $P(n)$ diamonds to un-suspend the whole thing during the push and pop operations. Since both push and pop work at the front of the stack, we just need to be able to un-suspend enough of our data structure to expose the front, which will save on the amount of diamonds we need for these operations. This amount may of course depend on $P$ and $n$. With these two insights in mind, we lay out our desired interface.

\begin{definition}[Bounded Stack Interface]
Given $k \in \mathbb N$ and an element type $A$, a $k$-stack implementation is a type $S$ together with the following closed terms:
\begin{itemize}
  \item $\code{empty} : S$
  \item $\code{push} : \tparr{(\tplist{\tpunit})^k}{\tparr{\tptens{A}{S}}{\tptens{(\tplist{\tpunit})^k}{\tptens{S}{(\tpsum{A}{\tpunit})}}}}$
  \item $\code{pop} : \tparr{(\tplist{\tpunit})^k}{\tparr{S}{\tptens{(\tplist{\tpunit})^k}{\tptens{S}{(\tpsum{\tpunit}{A})}}}}$
\end{itemize}
\end{definition}

By $(\tplist{\tpunit})^k$ we mean the type of $k$-tuples $\tplist{\tpunit} \otimes \cdots \otimes \tplist{\tpunit}$. We use it here as a source of diamonds; the lists contain no information, but the diamonds stored alongside each element will be used to un-suspend parts of the data structure. The reason for using a $k$-tuple of lists instead of just one big list becomes apparent in the proof of \pref{lem:stack-inductive}, where the use of such a tuple greatly simplifies the implementation.

Intuitively, both $\code{push}$ and $\code{pop}$ take in some diamonds and the return all of them as their first return value. They both also return a stack and a sum as their second and third return values. The left branch of the sum is intended to signal failure of the operation (in which case the input stack is returned unmodified), and the right branch signals success (in which case the returned stack has been successfully modified). For $\code{push}$, which takes in an element and a stack, failure means that the input stack is full, and so the input element is returned to us. For $\code{pop}$, which just takes in a stack, failure means the input stack is empty. If the input stack has at least one element, it is returned to us via the right injection.

To formalize this intuitive specification, we use the denotational semantics. We first define the element $m_{n,k} \in \llbracket (\tplist{\tpunit})^k \rrbracket$ to be the tuple $(\ell, \ldots, \ell)$, where $\ell$ is the unique element of $\tpsem{\tplist{\tpunit}}$ such that $\ell = n$. This way, an \lfpl term representing $m_{n,k}$ contains exactly $nk$ diamonds.

\begin{definition}[Bounded Stack Correctness]
\label{def:stack-correctness}
Let $k \in \mathbb N$. Suppose $(S, \code{empty}, \code{push}, \code{pop})$ is a $k$-stack implementation with element type $A$. Given a function $B : \mathbb N \to \mathbb N$, we say this implementation is correct and bounded by $B$ if the following holds for all $n \in \mathbb N$:
\begin{itemize}
  \item There exists a unary relation $V \subseteq \tpsem{S}$. This is intended to represent the valid states of the data structure.
  \item There exists a function $I : \tpsem{S} \to \tpsem{A}^*$. This is intended to represent the list of items contained in the stack.
  \item Let $s = \tmsem{\code{empty}}{\ctxnil}$. Then, $V(s)$ holds and $I(s) = []$.
  \item Let $f = \tmsem{\code{push}}{\ctxnil}$. Then, for all $x \in \tpsem{A}$ and $s \in \tpsem{S}$ such that $V(s)$ holds, we have:
  \begin{itemize}
    \item If $|I(s)| = B(n)$, then $f(m_{n,k})(x, s) = (m_{n,k}, s, (1, x))$.
    \item If $|I(s)| < B(n)$, then $f(m_{n,k})(x, s) = (m_{n,k}, s', (2, \ast))$, where $V(s')$ holds and $I(s') = x :: I(s)$.
  \end{itemize}
  \item Let $f = \tmsem{\code{pop}}{\ctxnil}$. Then, for all $s \in \tpsem{S}$ such that $V(s)$ holds, we have:
  \begin{itemize}
    \item If $I(s) = []$, then $f(m_{n,k})(s) = (m_{n,k}, s, (1, \ast))$.
    \item If $I(s) = x :: \ell$, then $f(m_{n,k})(s) = (m_{n,k}, s', (2, x))$, where $V(s')$ holds and $I(s') = \ell$.
  \end{itemize}
\end{itemize}
\end{definition}

The intended use of the data structure is to choose $n \in \mathbb N$ and always pass $m_{n,k}$ to the $\code{push}$ and $\code{pop}$ operations. In other words, we commit to a value of $n$ at runtime and always provide $nk$ diamonds to the stack operations. \pref{def:stack-correctness} says that, in this use case, the stack can hold at most $B(n)$ items. Behavior is otherwise undefined. Note that whenever $\code{push}$ and $\code{pop}$ receive $m_{n,k}$ as input, they return $m_{n,k}$ as the first element of their output. This way, no diamonds are ever permanently used, only temporarily borrowed, and thus available for future stack operations.

The definition only makes claims about valid stacks, i.e. $s \in \tpsem{S}$ such that $V(s)$ holds. Naturally, we insist that $\code{empty}$ should always be valid, and that $\code{push}$ and $\code{pop}$ should preserve validity. This extra layer of complexity is necessary; many of the implementations we provide in \pref{sec:completeness-bounded-stack-implementation} will require stack types $S$ which contain elements that would cause the $\code{push}$ and $\code{pop}$ operations to fail their specification. \comment{Nathan}{Potential connection to Atkey's dependent LFPL here?}

The intuition for $\code{push}$ is that, given an element to push $x \in \tpsem{A}$ and a valid input stack $s \in \tpsem{S}$, we case on whether $s$ is full, i.e. $|I(s)| = B(n)$. If it's full, we just return $s$ and $x$. Otherwise, we return some new valid stack $s'$ such that the items of $s'$ are $x$ followed by the items of $s$. The intuition for $\code{pop}$ is quite similar, casing on whether $I(s)$ is $[]$ or $x :: \ell$ and proceeding accordingly.

To summarize, a correct $k$-stack bounded by $B : \mathbb N \to \mathbb N$ is a data structure which, given the user's choice of some number $n \in \mathbb N$ at runtime, behaves like a standard stack in a functional programming language but requires temporary access to $nk$ diamonds and has a storage limit of $B(n)$ elements.



\subsection{Bounded Stack Implementations}
\label{sec:completeness-bounded-stack-implementation}

We now provide a concrete implementation of a bounded stack and several ways to build better implementations on top of pre-existing ones. These can ultimately be combined to give us a stack bounded by our choice of a polynomial $P$, which is exactly what we want for simulating the tape of a polynomial-time Turing machine. While most of the proofs in this section focus on the high-level intuition and avoid writing or reasoning about concrete \lfpl programs, everything is formalized in the mechanization (see \pref{sec:mechanization-soundness-completeness}).

\begin{lemma}[Constant Stack Construction]
\label{lem:stack-constant}
For any element type $A$ and $c \in \mathbb N$, there exists a correct $0$-stack implementation which is bounded by $B(n) = c$.
\end{lemma}
\begin{proof}
We take the implementation type to be $S = (1 + A)^c$. Intuitively, a left injection represents an open space in the stack, and a right injection represents a stored element. A stack $(x_1, \ldots, x_c) \in \tpsem{S}$ is considered valid when there is some $0 \le i \le c$ such that $x_j = (1, \ast)$ for $1 \le j \le i$, and $x_j$ is a right injection for $i < j \le c$.

To find the head of the stack, find the first $i$ such that $x_i = (2, a)$ for some $a \in \tpsem{A}$; the value $a$ is the head element. If $i$ is $1$, then the stack is full. If no such $i$ exists, the stack is empty. Since $c$ is a known constant, this procedure of locating the stack head does not require \lfpl iteration, so indeed no diamonds are required for the $\code{push}$ and $\code{pop}$ operations.
\end{proof}


\begin{lemma}[Inductive Stack Construction]
\label{lem:stack-inductive}
Fix an element type $A$ and some $k \in \mathbb N$. Suppose $(S, \code{empty}, \code{push}, \code{pop})$ is a $k$-stack implementation with element type $A$ that is correct and bounded by $B : \mathbb N \to \mathbb N$. Then, there is a $(k + 1)$-stack implementation with element type $A$ that is correct and bounded by $B'(n) = n B(n)$. 
\end{lemma}

\end{proof}


\begin{lemma}[Monomial Stack Construction]
\label{lem:stack-monomial}
Given $c,k \in \mathbb N$, there exists a $k$-stack implementation with element type $A$ that is correct and bounded by $B(n) = cn^k$.
\end{lemma}
\begin{proof}
Going by induction on $k$, \pref{lem:stack-constant} and \pref{lem:stack-inductive} are precisely the base and inductive cases we need, respectively.
\end{proof}


\begin{lemma}[Weakened Stack Construction]
\label{lem:stack-weakened}
Fix an element type $A$ and some $k \in \mathbb N$. Suppose $(S, \code{empty}, \code{push}, \code{pop})$ is a $k$-stack implementation with element type $A$ that is correct and bounded by $B : \mathbb N \to \mathbb N$. Then, there is a $(k + 1)$-stack implementation with element type $A$ that is correct and bounded by $B$. 
\end{lemma}
\begin{proof}
This construction is relatively obvious; if $\code{push}$ and $\code{pop}$ need only $kn$ diamonds, then instead having to accept $(k + 1)n$ diamonds is no problem. We can reuse the implementation type $S$ as well as the code for $\code{empty}$. For the new stack operations $\code{push}'$ and $\code{pop}'$, they simply make use of the fact that $m_{n,k+1} = (\ell,m_{n,k})$ for some $\ell \in \tpsem{\tplist{\tpunit}}$. They just ignore $\ell$ and call $\code{push}$ and $\code{pop}$ (respectively) with $m_{n,k}$.
\end{proof}


\begin{lemma}[Additive Stack Construction]
\label{lem:stack-additive}
Fix an element type $A$ and some $k \in \mathbb N$. For $i \in \{1,2\}$, suppose $(S_i, \code{empty}_i, \code{push}_i, \code{pop}_i)$ is a $k$-stack implementation with element type $A$ that is correct and bounded by $B_i : \mathbb N \to \mathbb N$. Then, there is a $k$-stack implementation with element type $A$ that is correct and bounded by $B(n) = B_1(n) + B_2(n)$. 
\end{lemma}
\begin{proof}
We take the implementation type to be $S = \tptens{S_1}{S_2}$. A stack $(s_1, s_2) \in \tpsem{S}$ is considered valid when $s_1$ and $s_2$ are valid, and $s_1$ is only nonempty when $s_2$ is full. Intuitively, $s_1$ contains the first section of our stack and $s_2$ contains the second. The operations are very similar to that of \pref{lem:stack-constant} with $c = 2$. To pop the stack, we first try popping $s_1$. If that fails, it must've been empty, so we pop $s_2$. To push some $x \in \tpsem{A}$ onto this stack, we first try pushing to $s_2$. If that fails, it means $s_2$ is full, so we push to $s_1$.
\end{proof}


\begin{lemma}[Polynomial Stack Construction]
\label{lem:stack-polynomial}
Given a polynomial $P : \mathbb N \to \mathbb N$ with degree $k$, there exists a $k$-stack implementation with element type $A$ that is correct and bounded by $P$.
\end{lemma}
\begin{proof}
Write $P(n)$ as $\sum_{i=0}^k c_i n^i$. For each $0 \le i \le k$, we can use \pref{lem:stack-monomial} to obtain an $i$-stack with bound $B_i(n) = c_i n^i$. Then, we can repeatedly use \pref{lem:stack-weakened} to weaken all of them into $k$-stacks with the same bounds. Finally, we can repeatedly use \pref{lem:stack-additive} to obtain the desired $k$-stack whose bound is the sum of the bounds of the weakened monomial stacks.
\end{proof}
